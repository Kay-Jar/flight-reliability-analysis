## Please list out changes in the directions of your project if the final project is different from your original proposal (based on your stage 1 proposal submission).

- Compared to our original proposal, the main direction of FRAA stayed the same but we had to reduce the scope of some proposed features. Our original goal was to build a flight data analysis tool using the Bureau of Transportation Statistics (BTS) datasets with filters for airlines, airports, dates, and delay types. The final project still does this by allowing users to query the database and view results through tables and a heatmap to visualize.
- One major change was removing the map visualization. In the proposal, we planned to use OpenStreetMaps and/or Leaflet but we did not implement this simply due to time and the complexity. Instead, we focused on making sure the database can query and the heatmap visualization worked.
- The set of filter axes also shifted. The original proposal named airlines, airports, dates, and delay types as the four filter dimensions, but the final application drops the date-range filter entirely (the loaded BTS snapshot is treated as a single time window) and instead introduces two new carrier-centric filters: an Operating Airline filter that is distinct from the marketing carrier, and a Carrier Delay Tier filter (Excellent / Good / Average / Poor) computed against the fleet-wide average arrival delay. This pushed the application toward carrier-reliability analysis rather than time-series exploration.
- We also shifted away from building a full dashboard with many features and visualizations and instead focused on a final application that focused on the database filtering and analytical queries. This made us prioritize the required features, such as the CRUD ops and advanced database functionality.
- Notably, we also switched the database engine from PostgreSQL to MySQL. The production deployment now runs on a GCP Cloud SQL (MySQL) instance accessed by a GCP Compute Engine VM that hosts the FastAPI backend and the built frontend bundle.
- Relatedly, the deployment model itself shifted away from the Dockerized multi-container design described in the proposal. We originally planned to use Docker Compose to orchestrate PostgreSQL, the backend, and the frontend on a single host, but the final architecture uses managed Cloud SQL for the database and a single GCP VM for the application tier. Docker Compose is still present in the repo for local development against a Postgres instance, but it is no longer the primary deployment path.
- On the frontend tech stack, we built the UI with React and plain JSX rather than React with TypeScript as proposed. Given the relatively small UI surface area and the team's stronger Python/SQL focus, the type-safety win did not justify the added overhead, and we leaned on the CoreUI React admin template instead.
- As such, the project became more focused on having something reliable data-wise rather than a wide range of visualizations.

## Discuss what you think your application achieved or failed to achieve regarding its usefulness.

- Our application (FRAA) aims to solve the limited tools that allow end users to analyze data from the aviation industry. FRAA achieves this goal through data fetching and querying to display the resulting data based on the filters the user applied. Additionally, FRAA also generates a resulting heatmap of the filtered data for guided visualization.
- However, the application does not fully achieve all of the intended usefulness. Some of our planned features, like the map visualization, were not implemented simply due to time constraints. As a result, FRAA is still effective for querying and having basic visualization but it did not reach the full potential of our original proposal in terms of full visualization.
- Another unfortunate limitation is that our dataset is not continuously updated. Because FRAA cannot fetch live or recent flight information, it may not reflect the most current patterns or emerging trends in aviation data.

## Discuss if you changed the schema or source of the data for your application

- The source of our data did not change. Throughout the semester we kept using the same BTS datasets and TranStats Aviation metadata tables outlined in the proposal. The schema, however, did pick up a handful of targeted changes: partly to clean up the data so it actually fit the design, and partly to add the columns and tables we needed for new application features.
- One of which was filling in missing WAC codes in the dim_wac table. Other source tables referenced WAC codes that were not present in dim_wac, and since we designed our database to link tables via foreign key constraints on that attribute, we resolved this by inserting placeholder rows into dim_wac for those missing codes.
- Another thing is that we identified that cancelled and diverted flights were stored with a delay time that wasn’t NULL. This doesn’t make sense though since a flight like that technically shouldn’t have a delay at all. Our final dataset has all cancelled flights set their delay time to NULL, which we then filter out whenever calling queries involving delay time.
- We added op_carrier_airline_id to the fact_flight table with FK fk_flight_op_carrier → dim_carrier.airline_id, distinct from the existing mkt_carrier_airline_id (marketing carrier). Alongside it we added branded_code_share, a VARCHAR column on fact_flight that records the marketing/operating-carrier brand pairing per flight (relationship between the marketing and regional carriers, such as AA being the flight code shared between a regional operator and American Airlines). Both fields are exposed when the user requests the raw flight information.
- We layered a few CHECK constraints onto the schema for DB-level data integrity: chk_flight_distance_nonneg and chk_flight_air_time_nonneg on fact_flight, and chk_delay_minutes_positive on fact_flight_delay. These let the database itself reject obviously bad rows during loading, instead of relying on the loader scripts to be perfectly defensive.

## Discuss what you change to your ER diagram and/or your table implementations. What are some differences between the original design and the final design? Why? What do you think is a more suitable design?

Comparing our Stage 2 ER diagram to the final implementation, the structural shape of the database did not actually change much. The Stage 2 design was already a star schema, with fact_flight at the center, fact_flight_delay as the per-delay-type bridge, and dim_date, dim_wac, dim_airport, dim_carrier, dim_delay_type, and saved_preset as the surrounding tables. We did not collapse a wider table into smaller dim tables for the final version, that decomposition had already happened by Stage 2, so the differences between the original ER diagram and the final schema are narrow.

The first change was adding op_carrier_airline_id to fact_flight, with a second FK (fk_flight_op_carrier) into dim_carrier.airline_id distinct from the existing mkt_carrier_airline_id. The BTS data exposes a marketing-vs-operating-carrier distinction (a regional operator flying under a mainline brand), which the Stage 2 fact_flight collapsed into a single carrier reference. The second change was adding branded_code_share as a VARCHAR column on fact_flight to carry the codeshare brand label per flight. We think the final design is the more suitable one for FRAA. The marketing/operating split via the second carrier FK and the branded_code_share column give us far more distinction between airlines in the dataset and more meaninful comparisons to make between different carriers.

## Explain how you think your advanced database programs complement your application.

Our advanced database programs each back a specific user feature. sp_carriers_above_avg_delay groups fact_flight by carrier and returns only those whose average delay exceeds the fleet wide average, which are then highlighted for the user to avoid in the Carrier Summary table. sp_busiest_airports UNIONs two GROUP BY queries to return airport specific counts with a role label, feeding the Busiest Airports panel. sp_top_delay_routes first picks the top 10 routes by delay count, then joins fact_flight_delay and dim_delay_type to break those routes' total delay minutes down by delay type, feeding the Top Routes panel. sp_classify_carrier_delay_tiers uses a cursor to walk carriers with more than 100 flights and assign each one a tier (excellent, good, average, or poor). This drives both the tier badge in the Carrier Summary. All of these help the user identify which carriers are the best and worst for their upcoming trips.

## Each team member should describe one technical challenge that the team encountered.  This should be sufficiently detailed such that another future team could use this as helpful advice if they were to start a similar project or where to maintain your project.

### **Kayetan Jarzabek:**

One major challenge we encountered was trying to maintain some sort of consistency between our local database and our cloud database on GCP. Initially in our proposal, we planned to use PostgreSQL for the local database but when it came time to set up the GCP for the extra credit, we realized that our database for previous stages were set up using MySQL. This caused many issues and headaches because the SQL syntax differs between the two. Certain queries ended up not working on the GCP cloud deployment and so we had to resolve this by manually changing the syntax of our PostgreSQL db files into MySQL. For future teams, I would highly recommend to decide on one database early and focus on it so future teams don’t end up having deployment issues due to syntax and tech stack differences in regards to their database.

### **Indexing (Gokul Sriramasubramanian):**

Another big challenge we encountered was during the indexing part of the project. We were experimenting with various indexes in the database to determine which ones improved performance the most, and keeping the indexing experiments clean and isolated when testing each design in the MySQL database was complicated and needed some workarounds. Basically, once we created an index, sometimes we weren’t able to drop it because MySQL reused them to enforce foreign key constraints. This meant that later tests sometimes ran in an environment where previously created indexes still existed, which could influence the explanation, analyze outputs and make direct comparisons between index designs difficult. To overcome this, we adjusted the testing methodology itself instead of relying on a perfectly clean reset for every test. We used “USE  INDEX” and “FORCE INDEX” inside queries to direct MySQL toward the specific index design we wanted to test, which let us compare different index designs even when other indexes were still in the database. For future teams, I would suggest planning out the index experiments in advance, document each and every index, test one query at a time, and use “USE  INDEX” and “FORCE INDEX” if cleanup is hard to do. For the most accurate results, creating a fresh copy of the database for each major experiment would be best, but that might be overkill and not actually reveal any new information.

### **Noah Zhang:**

One major (and really annoying) challenge I encountered was the dataset not being compatible with the relational schema and ER diagram we drafted. On paper, the attributes for each table matched perfectly and our FK relationships made perfect sense. The data however did not support this and we found missing and mismatched columns across our different tables, one specific example being the list of WAC codes being non-exhaustive across tables. Basically, rows of flight information references WAC codes not stored in our dim_wac table, and because there is a foreign key restraint on that, those rows are skipped when uploading to GCP or break the DB loading when constructing locally. Our solution for this was inserting placeholder rows wherever necessary. For future teams however, it would be best to fully analyze the data present inside the datasets you are using before designing the schema for your database.

### **Schema Design (Rachel Li):**

When building the schema for the data from the BTS, we realized that the original datasets were not structured in the most efficient way for aggregations and filtering, which would have made querying much more complex and less efficient over time. We solved this issue by researching different types of schema designs and decided to implement a STAR schema for our dataset, which helped make our joins successful and our queries easier to maintain. Because of this, a challenge was that this took up a lot of additional time and research to put into this project, while still also moving forward with the project. For future teams, I would recommend allotting time for updates because schema designs can often change and be updated after testing on real data, so allotting time for revisions can save time down the road.
